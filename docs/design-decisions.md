# Design Decisions

> 주요 설계 결정 사항과 그 이유를 기록합니다.

## 1. Flutter + Dart 채택
- **날짜**: 2026-04-04
- **결정**: Flutter 프레임워크로 개발
- **이유**:
  - 하나의 코드베이스로 Android/iOS 동시 지원
  - 빠른 프로토타이핑 (Hot Reload)
  - WebRTC 관련 패키지 생태계 존재 (flutter_webrtc)
- **대안**: Kotlin(Android 네이티브) → 크로스 플랫폼 지원을 위해 Flutter 선택

## 2. Riverpod 상태 관리 채택
- **날짜**: 2026-04-04
- **결정**: Riverpod을 상태 관리 솔루션으로 사용
- **이유**:
  - 컴파일 타임 안전성 (Provider보다 타입 안전)
  - 테스트 용이 (ProviderContainer로 독립 테스트)
  - Provider의 한계(BuildContext 의존) 해결
- **대안**: BLoC (Flutter 커뮤니티 인기) → MVVM 패턴과의 자연스러운 결합을 위해 Riverpod 선택

## 3. MVVM 아키텍처 채택
- **날짜**: 2026-04-04
- **결정**: MVVM 패턴으로 아키텍처 구성
- **이유**:
  - View/ViewModel/Model 분리로 관심사 명확
  - Riverpod Provider가 ViewModel 역할을 자연스럽게 수행
  - 이전 Android 프로젝트에서의 경험 활용 가능
- **대안**: BLoC 패턴 → 프로젝트 규모 대비 보일러플레이트가 많아 MVVM 선택

## 4. WebRTC 영상 전송 채택
- **날짜**: 2026-04-04
- **결정**: WebRTC를 P2P 영상 전송 기술로 사용
- **이유**:
  - 저지연 실시간 스트리밍 (CCTV 용도에 적합)
  - P2P 연결로 별도 미디어 서버 불필요 (LAN 환경)
  - 표준 프로토콜, flutter_webrtc 패키지 지원
  - 향후 외부 접속 확장 시 STUN/TURN만 추가하면 됨
- **대안**: RTSP/RTMP → 별도 미디어 서버 필요, 설정 복잡

## 5. Node.js 자체 시그널링 서버 구현
- **날짜**: 2026-04-04
- **결정**: Node.js + ws 패키지로 WebSocket 시그널링 서버를 자체 구현
- **이유**:
  - 시그널링 프로토콜에 대한 학습 목적
  - 외부 서비스 의존 없이 LAN에서 완전 독립 동작
  - 추후 기능 확장 시 자유도 확보
  - WebRTC + Node.js 레퍼런스/예제가 풍부
  - ws 패키지로 수십 줄이면 구현 가능
- **대안**: Dart 서버 → 생태계/레퍼런스 부족, Firebase → 외부 의존성/인터넷 필요

## 6. LAN 우선 개발 전략
- **날짜**: 2026-04-04
- **결정**: 같은 WiFi 네트워크(LAN) 환경을 우선 지원하고, 외부 접속은 추후 확장
- **이유**:
  - 핵심 기능(영상 스트리밍)에 집중
  - NAT 트래버설, TURN 서버 등 복잡도 제거
  - LAN에서 동작 검증 후 외부 접속 추가가 자연스러움
- **확장 계획**: STUN/TURN 서버 추가, 인증 레이어 추가

## 7. 단일 앱 + 모드 선택 방식
- **날짜**: 2026-04-04
- **결정**: 카메라 앱과 뷰어 앱을 분리하지 않고, 하나의 앱에서 모드를 선택
- **이유**:
  - 배포/설치 편의성 (하나의 APK)
  - 코드 재사용 극대화 (WebRTC, 시그널링 로직 공유)
  - 두 폰에 같은 앱 설치 후 역할만 선택
- **대안**: 카메라/뷰어 별도 앱 → 중복 코드 증가, 관리 포인트 2배
